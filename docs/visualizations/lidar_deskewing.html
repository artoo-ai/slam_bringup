LiDAR Deskewing

Deskewing (motion compensation / undistortion) corrects the geometric smear that happens because a LiDAR scan is not an instantaneous snapshot. The Livox Mid-360 takes ~100 ms per sweep. During that window the robot moves, so the first point in the scan and the last point were measured at different robot poses. Without correction every point is plotted as if the robot were frozen — producing a "skewed" cloud that warps walls, doubles edges, and drifts the map. This page explains what deskewing is, when you need it, whether this repo currently does it, and how the relevant parameters affect the output.

Do We Deskew Now? 3D: yes (implicit) 2D: no

The Short Answer

Launch scriptPipelineDeskewing?How
start_slam.sh FAST-LIO2 (3D lidar-inertial) Yes FAST-LIO2 internally uses IMU samples inside each scan window to motion-compensate every point before ICP registration. This is built into the algorithm — there is no on/off flag.
start_slam_2d.sh pointcloud_to_laserscan + slam_toolbox No The 3D cloud is flattened to a 2D LaserScan at ~10 Hz. slam_toolbox treats each scan as a rigid frame and matches scan-to-scan. No per-point motion correction.

start_nav_2d.sh slam_toolbox localization + Nav2 No Same 2D pipeline — no deskewing.
Key insight: FAST-LIO2's deskewing is implicit and always-on. It uses the Mid-360's onboard ICM40609 IMU (hardware time-synced to the LiDAR) to predict where the robot was at each point's exact timestamp, then transforms that point back to a consistent reference pose. There is no deskew: true flag — it's baked into the ESIKF state estimation loop.
2D stack doesn't need it (yet): At indoor mecanum-chassis speeds (< 0.5 m/s), the Mid-360's 100 ms sweep causes < 5 cm of smear in the flattened 2D scan. slam_toolbox's scan matching easily absorbs this. You would need deskewing in the 2D stack if the rover moved > 1 m/s or rotated > 90 deg/s during mapping.

Other LiDARs: Pandar40P & VLP-16 no built-in IMU

Mid-360 vs Mechanical Spinning LiDARs

The Mid-360's deskewing story is simple because its IMU lives on the same board. The Pandar40P and VLP-16 are mechanical spinning LiDARs with no onboard IMU — everything changes.

No built-in IMU = you must provide one. The Pandar40P and VLP-16 have no onboard IMU. To use FAST-LIO2 (or any lidar-inertial SLAM), you need an external IMU rigidly mounted near the LiDAR. Candidates: WitMotion WT901C, Bosch BMI088, or any 6-axis IMU at ≥ 200 Hz. You must then:
  • Set time_sync_en: true
  • Measure time_offset_lidar_to_imu (record simultaneous data, cross-correlate timestamps)
  • Physically measure and set extrinsic_T (IMU-to-LiDAR translation in meters)
  • Calibrate extrinsic_R (IMU-to-LiDAR rotation — use extrinsic_est_en: true for one run, then lock)
VLP-16 on the rotating platform — double deskewing: If you spin the VLP-16 on the turntable to extend its 30° vertical FOV to full-spherical, the platform rotation adds continuous motion on top of robot motion. You need two stages of deskewing:
  1. Encoder-based deskew — transform each point from the rotating sensor frame to the stationary base frame using the AS5600 encoder angle at that point's timestamp. This is the live deskew node from the rotating LiDAR build notes.
  2. IMU-based deskew — FAST-LIO2 (or KISS-ICP) then corrects for robot motion during the sweep, as usual.
Without stage 1, SLAM sees the world doing backflips and instantly diverges. Without stage 2, you get the normal motion smear on top of the corrected cloud.
Alternative: IMU-free SLAM. If you don't want to deal with IMU clock sync, CT-ICP and KISS-ICP can deskew without an IMU — CT-ICP uses continuous-time elastic ICP (linear interpolation between start/end pose), KISS-ICP uses a constant-velocity model. Both work well at vehicle speeds but struggle with aggressive rotation (> 90 deg/s). For the Pandar40P on a slow-moving rover this is a viable option.

The Math (simplified)

For each point pᵢ measured at time tᵢ within the sweep [t₀, t₁]:
  1. Use IMU gyroscope + accelerometer to estimate the robot's pose change ΔT(tᵢ) from the reference time to tᵢ
  2. Compute the rigid transformation: p_corrected = ΔT(tᵢ)⁻¹ · pᵢ
  3. The corrected point is now in the reference frame as if all points were measured simultaneously
Why the IMU must be hardware time-synced: The correction depends on knowing exactly which IMU sample corresponds to which LiDAR point. The Mid-360's onboard ICM40609 shares a clock domain with the LiDAR firmware — timestamps align to < 1 ms. An external IMU (like the WitMotion WT901C) uses ROS-stamped host time, which can drift 5–10 ms. That's enough skew to warp the deskewed cloud and introduce drift. This is why fast_lio_mid360.yaml subscribes to /livox/imu_ms2 (the unit-corrected Mid-360 IMU), not /imu/data (WitMotion).

When You Need Deskewing decision guide

Smear Budget

The amount of geometric error (smear) from skipping deskewing depends on two things: robot speed and sweep duration.

smear = velocity × sweep_time

SpeedSweep (100 ms)SmearImpact
0.1 m/s (slow indoor)100 ms1 cmNegligible — skip deskewing
0.3 m/s (walking pace)100 ms3 cmMinor — 2D scan matching absorbs it
0.5 m/s (fast indoor)100 ms5 cmVisible in 3D cloud — walls thicken
1.0 m/s (hallway cruise)100 ms10 cmSignificant — 3D SLAM needs deskewing
2.0 m/s (outdoor rover)100 ms20 cmSevere — map corrupts without deskewing

Rotation is worse: At 90 deg/s yaw (a quick turn), points 100 ms apart are rotated 9 degrees from each other. A wall 3 m away smears by 3 × sin(9°) ≈ 47 cm.

Rule of thumb: If max smear > half your voxel/grid resolution, you need deskewing. FAST-LIO2's filter_size_surf: 0.3 means deskewing matters above ~15 cm smear (> 1.5 m/s translation or > 60 deg/s rotation at 3 m range).

IMU-LiDAR Clock Alignment why time_sync_en stays false

The Three IMUs on This Rover

IMUTopicClock SourceDeskew-Safe?Used By
Mid-360 ICM40609 /livox/imu_ms2 Hardware-synced to LiDAR (shared FPGA clock) Yes FAST-LIO2 (deskewing + state estimation)
D435 BMI085 /d435_front/.../imu Hardware-synced to camera (shared USB frame clock) No — wrong clock domain Not used in current stack
WitMotion WT901C /imu/data ROS host timestamp (software-stamped on arrival) No — 5–10 ms jitter Backup / attitude reference
The unit-correction trap: The Livox driver publishes /livox/imu with acceleration in units of g (1.0 at rest). FAST-LIO2 expects m/s² (9.81 at rest). Subscribing to the raw topic silently degrades pose estimation — gravity init reads 1.0, acc covariance is over-weighted by ~96x, and the IMU prediction step contributes effectively nothing to deskewing. The imu_units_node.py republisher multiplies by 9.80665 and publishes /livox/imu_ms2. Always subscribe to the corrected topic.
Diagnostic check: Run lidar_diagnostics.py — it monitors IMU↔LiDAR timestamp offsets and warns if > 50 ms. If you see that warning, check that you're using the Mid-360's built-in IMU, not the WitMotion.

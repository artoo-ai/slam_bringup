SLAM Config Explorer

Every interactive card below maps to a real parameter in this repo's config files. Slide it, watch the data change, then decide whether you want to push that knob up or down. The "tune up if / down if" lines at the bottom of each card describe the trade-off in terms of what you'd actually see on the robot — drift, missed obstacles, planner indecision, CPU load. Companion to docs/visualizations/slam_2d_vs_3d.html. Each card shows the project default (from the YAML / launch file) next to every slider — green when current = default, yellow when you've moved away. Use ↺ Reset per card, or the button on the right to revert everything.

What changes: how finely the 360° around the robot is sliced. Each bin reports one min-range value.
↑ Larger °/bin: fewer rays, less CPU, but a chair leg can hide behind the wall behind it ("angular aliasing").

The three "rings" around the rover — sensor vs planner vs cost how range_min · robot_radius · inflation_radius differ All three look like circles around the robot, but they live in different stages of the pipeline and answer different questions. This card draws them all on one scene so you can see who blocks what.

Ring Lives in Hard or soft? The question it answers
scan_range_min / blind sensor driver / pre-SLAM
(launch + fast_lio)		 HARD exclude
(returns dropped before SLAM sees them)		 "Is this point even a real obstacle, or is it the rover hitting itself?" Filters out sensor returns from the rover's own bumper, fixture, antennas. Has nothing to do with planning.
robot_radius Nav2 costmap
(LETHAL inflation)		 HARD forbid
(cells inside this distance of any obstacle = LETHAL → planner refuses)		 "How big am I, physically?" Any cell whose center is within robot_radius of an obstacle is treated as if the robot is crashing into it. The planner won't put the robot's center there. Match the actual bumper.
inflation_radius Nav2 costmap
(inflation layer)		 SOFT cost
(decaying gradient outside robot_radius)		 "How much should I prefer to stay away?" Outside the lethal zone, costs decay smoothly via exp(-K·(d − robot_radius)). The planner will still cross inflated cells if there's no alternative, but prefers the cheaper middle of a corridor. Tunes style, not legality.
Watch as you slide: the yellow dashed ring (range_min) discards returns near the rover before they enter the map; the cyan solid ring (robot_radius) is a hard wall the path can't cross; the red gradient (inflation) bends the path toward open space without forbidding closer cells.
🔗 They stack, they don't replace each other: a sensor return survives range_min → it becomes an obstacle cell in the costmap → that cell gets a lethal ring of width robot_radius → which then gets an inflation gradient of width inflation_radius on top. So the total "no-go" distance from a true obstacle is robot_radius + inflation_radius (with the inner part hard-forbidden and the outer part soft-penalized).

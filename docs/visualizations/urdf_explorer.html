1 · URDF Fundamentals

A URDF is a tree of links (rigid bodies) connected by joints (relationships between bodies). Every link can have <visual>, <collision>, and <inertial> children. Joints have an <origin> (where the child sits in the parent's frame) and a type (fixed, revolute, continuous, prismatic, planar, floating). Start here if you've never written XML for a robot.

<robot name=…>root tag · *.urdf.xacro
fundamentals beginner name: kebab-case-string
The outermost wrapper. name is the robot's identifier — used by robot_state_publisher as a TF prefix in some setups, but mostly cosmetic. The xmlns:xacro attribute is required only if you use xacro features.
<robot xmlns:xacro="http://www.ros.org/wiki/xacro" name="mecanum_x3"> <!-- everything else goes inside --> </robot>
Where usedThis file's name="mecanum_x3", name="bench_fixture", name="sensor_rig". Each platform xacro has a unique name so they don't collide if loaded together.
Quick checkxacro urdf/mecanum.urdf.xacro | head -5 — confirm the root tag expands without error. The fastest sanity check that a xacro file is syntactically valid.
<link name=…>core element
fundamentals beginner unique within robot
A rigid body. Can be empty (just a coordinate frame — used heavily in this repo for sensor reference points like livox_frame, imu_link) or carry <visual> + <collision> + <inertial> children.
<!-- pure frame: TF node only, nothing drawn --> <link name="livox_frame"/> <!-- frame with a visual box --> <link name="base_link"> <visual>…</visual> </link>
Why empty links existEvery sensor in this repo needs a TF frame so ROS nodes (FAST-LIO2, realsense, witmotion) can publish data tagged with the correct frame_id. The frame doesn't need a mesh — the link element just declares "this name is a valid TF node."
Test planLoad the URDF in RViz with RobotModel display + TF display. Toggle individual link visibility. Confirm every sensor topic's frame_id matches a link defined here (ros2 topic echo /scan --once).
<joint name=… type=…>core element
fundamentals beginner parent ≠ child, no cycles
Defines the spatial relationship between two links. Has a parent and a child; the child sits at the joint's <origin> in the parent's frame. The URDF tree is a DAG — every link except the root has exactly one parent.
<joint name="base_link_to_sensor_plate" type="fixed"> <parent link="base_link"/> <child link="sensor_plate"/> <origin xyz="0 0 0.207670" rpy="0 0 0"/> </joint>
Tree constraintIf you try to make link A the child of link B AND link B the child of link A, robot_state_publisher will refuse to launch. Same if you forget to give a non-root link a parent — it becomes orphan and won't appear in TF.
joint type=…joint attribute
fundamentals intermediate 6 valid values
Picks the kinematic relationship. This repo uses only fixed because every sensor is rigidly bolted to the plate. The other types matter the moment you add wheels, a gripper, or a pan-tilt.
Cheat sheet fixed = bolted (no motion) · revolute = hinge with limits · continuous = wheel (unbounded rotation) · prismatic = linear slide with limits · planar = 2-DOF translation in a plane · floating = full 6-DOF (rarely used).
Test planWhen you mount a Yahboom chassis and want wheels to spin in RViz: add continuous joints with axis xyz="0 1 0" (rotation around Y) and publish wheel positions via joint_state_publisher. With fixed joints, RViz wheels stay glued.
<origin xyz="…" rpy="…"/>joint / visual / collision / inertial child
fundamentals REP-103 beginner meters · radians · ZYX intrinsic
The transform from the parent frame to the child frame. xyz = translation in meters (always). rpy = roll/pitch/yaw in radians (always), applied in ZYX intrinsic order (yaw first, then pitch around the new Y, then roll around the new X). Trip wire: many people assume degrees — URDF never accepts degrees.
Order mattersURDF rpy is ZYX intrinsic: rotate yaw around Z, then pitch around the rotated Y, then roll around the doubly-rotated X. If you write rpy="1.57 0 1.57" and rpy="0 0 1.57" then rpy="1.57 0 0", you get the SAME final orientation. But mid-computation the rotation axes differ — important when composing transforms manually.
Common bugWriting pitch in degrees. rpy="0 10 0" means 10 radians = ~573°. Always convert: 10° × π/180 = 0.1745. The D435 mount in this repo uses 0.176136 which encodes exactly 10.09° nose-down.
<geometry> primitivesvisual / collision child
fundamentals beginner box · cylinder · sphere · mesh
What shape to render. box size="L W H" (meters, axis-aligned, centered on origin). cylinder radius=… length=… (axis along +Z by default). sphere radius=…. mesh filename="package://pkg/path/to/file.stl" for STL/Collada meshes.
<geometry> <box size="0.280 0.230 0.090"/> </geometry>
Visual vs collisionUse detailed meshes for <visual> (eye candy in RViz), simplified primitives for <collision> (physics + Nav2 footprint). This repo only has <visual> boxes — collision footprints come from Nav2's robot_radius, not from URDF collision elements.
Test planTo swap the bench-fixture wireframe for a real STL: download an extrusion CAD, save as cad/2040_segment.stl, replace <box> with <mesh filename="package://slam_bringup/cad/2040_segment.stl"/>, rebuild + reload RViz. Geometry is now photoreal but SLAM math is unchanged.
<visual> + <material>cosmetic render
fundamentals beginner rgba 0..1 · ≤1 visual per material name
What RViz draws for this link. Has its own <origin> (where the visual sits relative to the link's frame) and <material> (color or texture). RViz only shows visual; it ignores collision elements unless you toggle the collision display.
<visual> <origin xyz="0 0 0.045" rpy="0 0 0"/> <geometry><box size="0.28 0.23 0.09"/></geometry> <material name="chassis_x3"> <color rgba="0.15 0.15 0.18 0.6"/> </material> </visual>
Alpha is your friendSet rgba "… … … 0.3" on extrusion boxes so you can see sensor positions through the chassis. This repo's bench_fixture.urdf.xacro uses 0.15–0.3 alpha for exactly this reason.
<collision>physics / contact
fundamentals intermediate used by Gazebo · ignored by Nav2 footprint
Geometry used for physics simulation and contact detection (Gazebo, MoveIt). For Nav2 navigation this repo bypasses URDF collision entirely — the footprint comes from Nav2's robot_radius param (see nav2_params_2d.yaml). If you're not simulating in Gazebo, skip collision elements entirely. They cost nothing to leave out.
Why this repo omits collisionNav2 reads its footprint from nav2_params_2d.yaml::robot_radius: 0.22, not from URDF. Simulation isn't part of the SLAM workflow on this rig — physical robot only. Adding collision elements does nothing here. If you ever spin up Gazebo for regression testing, that's when you wire them in.
<inertial> mass / inertiaphysics sim only
fundamentals advanced kg · kg·m² · symmetric tensor
Mass + inertia tensor for dynamics. Required by Gazebo to simulate forces; ignored by robot_state_publisher and RViz. <mass value="…"/> in kg. <inertia ixx iyy izz ixy ixz iyz/> in kg·m² — symmetric, so off-diagonals share signs.
<inertial> <mass value="1.2"/> <inertia ixx="0.005" iyy="0.012" izz="0.014" ixy="0" ixz="0" iyz="0"/> </inertial>
Estimating inertiaFor a box of mass m and dims (L, W, H): ixx = m/12 (W² + H²), iyy = m/12 (L² + H²), izz = m/12 (L² + W²). For a cylinder of mass m, radius r, length L (axis = Z): ixx = iyy = m/12 (3r² + L²), izz = m/2 · r². For meshes — load STL into Fusion 360 or MeshLab, set material density, export the computed inertia tensor.
Skip unlessYou're running Gazebo. Otherwise leave inertial blocks out — they're noise in the file and the SLAM stack ignores them.

2 · Coordinate Conventions (REP-103)

ROS standardizes axes: +X = forward, +Y = left, +Z = up. RPY follows the right-hand rule, so positive pitch (around +Y) tilts +X toward -Z — which means positive pitch = nose-down in URDF (NOT aerospace convention). Getting this wrong is the most common URDF bug.

REP-103 body axesconvention
REP-103 beginner +X fwd · +Y left · +Z up
The body frame is defined relative to the robot's "front" — in this repo, the direction the D435 camera looks. +X is forward, +Y is left, +Z is up. Visualize with the right hand: thumb = +X, index = +Y, middle = +Z. This is the same convention as REP-105 for the base_link frame.
Drag the canvas to rotate. Red = +X (forward). Green = +Y (left). Blue = +Z (up).
Why this mattersThe Mid-360 cable port, the D435 lens, and the WT901 X-axis sticker all have to point the SAME physical direction as URDF +X. If the cable port faces backward, every map will be mirrored. Confirmation step in this repo: sensors_common.urdf.xacro §"Sensor axes confirmed (2026-04-25)".
rpy sign conventionright-hand rule
REP-103 intermediate +pitch = nose-DOWN in URDF
URDF rpy follows the right-hand rule, not aerospace. +roll = right side dips down. +pitch = nose dips down (because R_y(+θ) takes +X toward -Z). +yaw = robot rotates counter-clockwise viewed from above. Slide the pitch slider — see how the camera FOV tilts.
Aerospace mismatchPilots write "+10° pitch up." URDF math says: R_y(+0.176) takes +X (forward) toward -Z (down) → camera looks at the floor → "nose-down." Same number, opposite intuition. The D435 mount is angled at +0.176136 rad = +10.09° which in URDF terms means nose-down.
Test planRun scripts/measure_d435_pitch.py with rig on a level floor. Inlier fraction ≥ 99% means the plane fit is reliable. Compare reported angle to d435_front_rpy in sensors_common.urdf.xacro. If they differ by more than ~0.5°, re-tighten the camera mount and remeasure — the foam-mount drift is real.
deg ↔ rad conversionutility
REP-103 beginner URDF rpy is ALWAYS radians
Slide a degree value, get the radian value you need to paste into URDF. Common reference points: 90° = π/2 ≈ 1.5708, 180° = π ≈ 3.14159, 10° ≈ 0.1745. This repo's D435 mount measured at 10.09° lives in URDF as 0.176136.
# Python helpers import math math.radians(10.09) # → 0.17611 math.degrees(0.176136) # → 10.09
CAD frame vs body framesensors_common.urdf.xacro · cad/make_livox_mid360_plate.py
REP-103 platform advanced 180° yaw mismatch
The lidar plate's CAD file uses a different axis convention than the URDF body frame. In CAD: +Y runs from WT901 → cable port. In URDF body: +X = D435 forward. They're rotated 180° in yaw. So plate -Y → body +X, plate +X → body +Y. Every xyz in sensors_common.urdf.xacro is pre-rotated into body convention so you can read them as "forward / left / up" directly.
The conversion table plate +X → body +Y (left) · plate -X → body -Y (right) · plate +Y → body -X (back) · plate -Y → body +X (forward) · plate +Z → body +Z (up, unchanged).
Trap to avoidWhen you add a new sensor, do NOT copy its coordinate from the CAD file directly. Either (a) re-derive in body frame from the start, or (b) note the CAD coordinate as a comment and write the body coordinate as the actual xacro property. The current xacros use option (b) — see comments in sensors_common.urdf.xacro for each sensor.

3 · xacro — Macros & Includes

URDF is pure XML, which means no variables, no math, no reuse. xacro is a preprocessor that adds ${expressions}, xacro:property (named constants), xacro:include (file composition), and xacro:macro (reusable templates). Run xacro file.urdf.xacro at launch time → emit pure URDF → feed to robot_state_publisher.

xacro:propertynamed constants
xacro beginner scalar · scope: file or include
A named constant. Use it for any value that appears more than once or that has physical meaning. Reference it with ${name}. Math works inside ${…}: ${a + b}, ${plate_thickness/2}, ${pi/4}.
<xacro:property name="plate_thickness" value="0.007"/> <xacro:property name="rubber_compressed" value="0.022987"/> <!-- compose: plate top - thickness/2 puts plate visual below origin --> <origin xyz="0 0 ${-plate_thickness/2}"/>
Single source of truthThis repo uses plate_thickness=0.007 in sensors_common.urdf.xacro. If you remill the plate to 5 mm, change one number and every derived xyz/origin updates. Same pattern for rubber_compressed, extrusion_2040_height, etc.
Test planOpen bench_fixture.urdf.xacro. Add <xacro:property name="extrusion_2040_height" value="0.050"/> (override). Re-run xacro + reload RViz. The whole stack-up grows 20 mm and base_link → sensor_plate auto-recomputes.
xacro:includefile composition
xacro beginner $(find pkg) path · 1 robot per file
Pulls another xacro file's contents inline. This repo uses it to share sensors_common.urdf.xacro (sensor mounting geometry) across every platform. The platform xacro defines base_link + chassis; the include adds the sensor plate + sensors on top.
<!-- bench_fixture.urdf.xacro --> <xacro:include filename="$(find slam_bringup)/urdf/sensors_common.urdf.xacro"/> <!-- now plate_thickness, sensor_plate link, livox_frame, imu_link, --> <!-- d435_front_link are all available in this file -->
$(find pkg) resolutionThe $(find slam_bringup) substitution is evaluated by xacro at parse time. It expands to the absolute path of the slam_bringup ROS package's share directory. As long as the package is in AMENT_PREFIX_PATH (true after source install/setup.bash), the include works from any working directory.
Test planRun xacro urdf/mecanum.urdf.xacro from a directory that hasn't sourced the workspace — expect "Cannot locate package 'slam_bringup'." Then source install/setup.bash and rerun — clean expansion.
xacro:macroparameterized templates
xacro intermediate named params · ${param} inside body
A reusable parameterized sub-tree. Define once, call N times with different arguments. This repo doesn't currently use macros (only 3 sensors, all single-instance), but you'd reach for one the moment you add 4 wheels or N camera mounts.
<xacro:macro name="wheel" params="prefix x y"> <link name="${prefix}_wheel">…</link> <joint name="${prefix}_wheel_joint" type="continuous"> <parent link="base_link"/> <child link="${prefix}_wheel"/> <origin xyz="${x} ${y} 0" rpy="0 0 0"/> <axis xyz="0 1 0"/> </joint> </xacro:macro> <xacro:wheel prefix="front_left" x="0.14" y="0.115"/> <xacro:wheel prefix="front_right" x="0.14" y="-0.115"/> <xacro:wheel prefix="rear_left" x="-0.14" y="0.115"/> <xacro:wheel prefix="rear_right" x="-0.14" y="-0.115"/>
When to add macros≥3 repetitions of similar geometry. mecanum.urdf.xacro will need one as soon as you add wheels for animated joint_states.
xacro:if / xacro:unlessconditional inclusion
xacro intermediate boolean expression
Conditionally include a block of XML based on a property or argument. Useful for "feature flags" inside a URDF — e.g. include a second D435 mount only on platforms that have one, or swap visual styles for sim vs real.
<xacro:arg name="with_rear_camera" default="false"/> <xacro:property name="with_rear" value="$(arg with_rear_camera)"/> <xacro:if value="${with_rear}"> <link name="d435_rear_link"/> <joint name="sensor_plate_to_d435_rear_link" type="fixed">…</joint> </xacro:if>
Test planLaunch with xacro urdf/mecanum.urdf.xacro with_rear_camera:=true. Verify the rear link appears in ros2 run tf2_ros tf2_echo sensor_plate d435_rear_link. Set false → same command returns "did not find transform."

4 · Frame Hierarchy & TF Tree

A URDF produces a tree of TF frames. The platform xacro defines base_link (the root). The shared sensors_common.urdf.xacro adds sensor_plate as a child, and livox_frame/imu_link/d435_front_link as grandchildren. At runtime FAST-LIO2 publishes camera_init → body separately; a static bridge connects body to base_link so the URDF tree and the SLAM tree merge into one connected graph.

REP-105 standard framesconvention
REP-105 intermediate map → odom → base_link → base_footprint
ROS standardizes four mobile-robot frames: map (global, drift-free, may jump at loop closure), odom (local, smooth, drifts over time), base_link (rigidly attached to the robot's body), base_footprint (projection of base_link onto the ground plane — Z=0). This repo uses the first three; base_footprint is only added on the mecanum platform if you wire it in for Nav2.
Who publishes whatmap → odom: slam_toolbox (mapping) OR localization (loc mode). odom → base_link: rf2o / FAST-LIO2 (via the body bridge). base_link → *: robot_state_publisher reads URDF and emits all static joints. Each transform has exactly one publisher — duplicates cause TF jitter.
When to add base_footprintNav2 footprint If the chassis isn't level (Go2 standing height varies), base_footprint stays at Z=0 while base_link rides up and down. Nav2's costmap reads base_footprint by default; without it on a quadruped, costmap clears below the dog.
parent ≠ child (tree constraint)URDF spec
fundamentals beginner no cycles · 1 parent per link
URDF is a directed acyclic graph rooted at base_link (or whatever link you don't list as a child anywhere). Every other link must be the child of exactly one joint. Trying to give a link two parents fails xacro expansion. Trying to make a cycle (A → B → A) fails robot_state_publisher startup.
"Closed loop" workaroundReal robots with closed kinematic chains (four-bar linkages, parallel manipulators) can't be expressed in URDF. You need SDF (Gazebo's format) or MJCF (MuJoCo). For this rig — purely tree-shaped — URDF is fine.

6 · Sensor Mounting (sensor_plate frame)

Every sensor's pose is expressed as an <origin xyz="…" rpy="…"/> relative to sensor_plate (top surface of the lidar plate, in body convention: +X = forward, +Y = left, +Z = up). Move a slider, see the sensor reposition in the side view. Don't approximate — sensor offsets directly shift the global map. A 5 mm error in the Mid-360 Y becomes a 5 mm bias in every wall position the SLAM stack reports.

Mid-360 (livox_frame)sensors_common.urdf.xacro · livox_xyz / livox_rpy
platform intermediate x: -0.025 · y: 0.0 · z: +0.03661 · rpy: 0 0 0
Body shifted 25 mm BACK (-X) to make room for the WT901 in front. Optical center sits 36.61 mm above the plate top per the Livox datasheet. Pure translation — no rotation needed because the Mid-360 was mounted with its +X aligned with body +X. Mid-360 publishes a 360° azimuth × ~59° elevation cloud, so a small body offset doesn't lose coverage.
0.03661 m is from the datasheet, not measuredThe Livox Mid-360 product manual specifies 36.61 mm from the mounting flange to the optical center. If you 3D-print a riser to lift the sensor higher, ADD the riser height to this value — don't replace it.
Other lidars Velodyne VLP-16: ~0.0717 m optical center above base Hesai Pandar40P: ~0.105 m RPLidar C1M1: ~0.040 m Unitree L1: ~0.027 m Each datasheet has the flange-to-optical-center number. Swap the sensor → swap this Z.
Test planPlace a 1 m vertical ruler against a wall, scan with rover stationary. In RViz, click a point on the ruler at the 0.5 m tick. Reported Z should be 0.5 m above floor (within ±5 mm). If it's off, this Z is wrong.
WT901 IMU (imu_link)sensors_common.urdf.xacro · imu_xyz / imu_rpy
platform intermediate x: 0.060 · z: +0.009 · rpy: 0 0 0
Plate-Y -60 mm in CAD frame → body +X 60 mm (toward D435). Z = 9 mm above plate top (3 mm flange + half of 12 mm housing-above-flange = housing center). Backup IMU only — FAST-LIO2 uses the Mid-360's onboard ICM40609. WT901 X-sticker physically points body-forward; no yaw correction needed.
If you rotate the WT901 housingThe X-axis sticker MUST stay aligned with body +X. If the housing is mounted sideways (sticker pointing left), set imu_rpy = "0 0 1.5708" to undo the 90° physical rotation. Otherwise the IMU reports yaw/pitch about the wrong axes and FAST-LIO2 diverges.
Other IMUs BNO086: similar 3 mm flange, ~10 mm housing BNO055: 4 mm flange, ~9 mm housing ICM-40609 (Mid-360 onboard): factory extrinsic [-0.011, -0.02329, 0.04412] from optical center, see fast_lio_mid360.yaml
Test planRun ros2 topic echo /imu/data while gently tilting rover nose-up. Pitch (Euler angle) should go POSITIVE per REP-103. If it goes negative, the WT901 is mounted upside-down or rotated — fix the rpy here, don't fix it downstream.
D435 front (d435_front_link)sensors_common.urdf.xacro · d435_front_xyz/rpy
platform advanced x: +0.147 · z: -0.029 · pitch: +0.176 rad (10.09° nose-down)
Front-glass plane is the URDF reference; realsense2_camera adds internal optical-center offsets at runtime. Sticking out 146.761 mm forward of plate center, 28.502 mm BELOW plate top (the mount drops it). Pitched +10.09° nose-down by design so depth FOV covers the floor ahead, not the horizon.
Don't trust the housing topThe D435 case rounds toward the lens, so resting an inclinometer on the top reads ~5° more pitch than the optical axis actually has. Use scripts/measure_d435_pitch.py — fits the floor plane in the depth cloud → reads the actual optical-axis tilt. The 10.09° in URDF came from a 99.8%-inlier floor fit on 2026-04-26.
How much pitch is "too much"At >25° nose-down, the D435 stops seeing the horizon entirely — usable for floor mapping but useless for obstacle avoidance at standing height. At <5° nose-down, depth coverage of the floor in front of the rover collapses to a thin band 2 m+ away; close-range floor obstacles get clipped by the lower FOV edge.
Other depth cameras D435i: same body as D435 + onboard IMU (publish to /imu/data) D455: wider baseline, ~95° FOV vs D435's 87° — re-derive pitch for similar floor coverage ZED Mini: 90° FOV — similar pitch OAK-D Pro: 80° FOV — needs MORE pitch (~15°) for same floor coverage

7 · PLATFORM_BRIDGES — body ↔ base_link

FAST-LIO2 publishes the transform camera_init → body at runtime, where body is the Mid-360 IMU body frame. The URDF tree's root is base_link. These two trees need to merge into one TF graph, which is the job of a single static_transform_publisher per platform — configured by the PLATFORM_BRIDGES dict in launch/slam.launch.py. This bridge is the most error-prone glue in the whole stack.

body → base_link offsetlaunch/slam.launch.py · PLATFORM_BRIDGES
runtime platform advanced bench_fixture: (0, 0, -0.244280) · mecanum: (0, 0, -0.144280)
For the bench fixture: base_link → plate_top = +0.207670, plate_top → livox_frame Z = +0.036610, so body sits 0.244280 m ABOVE base_link. The bridge transform goes body → base_link → so its Z translation is negative 0.244280. Get the sign wrong and the map appears inverted; get the magnitude wrong and the floor drifts up/down by exactly that error.
# launch/slam.launch.py PLATFORM_BRIDGES = { 'bench_fixture': (0.0, 0.0, -0.244280, 0.0, 0.0, 0.0), 'mecanum': (0.0, 0.0, -0.144280, 0.0, 0.0, 0.0), # 'go2': … ← fill in once measured # 'r2d2': … # 'roboscout': … }
The two-place update ruleIf you change base_link_to_plate_top in the platform xacro, you MUST change the Z in PLATFORM_BRIDGES[platform] in slam.launch.py too. Bridge Z = -(plate_top + livox_z_above_plate). Forgetting one of them leaves the URDF tree and the SLAM tree disagreeing by your mistake.
Test planAfter any stack-up change: place rover on level floor next to a known landmark (corner of a desk). Start mapping. In RViz, click the floor point in the map. z should be 0.0 m ± 5 mm. Click the desk's top edge — z should equal your desk's measured height. If the offset is constant, you forgot the bridge update.
Yaw mismatch (mecanum)launch/slam.launch.py · yahboom_bridge_node
runtime platform advanced FIX in bridge, NOT by rotating base_link
The YB-ERF01 firmware's +X is OPPOSITE the sensor "front" (D435 direction). The temptation is to add yaw=π to the URDF base_link to reconcile. Don't. That flips the Nav2 costmap, footprint orientation, and /scan frame 180° and breaks navigation. Instead the velocity-command inversion happens inside yahboom_bridge_node via invert_vx/invert_vy parameters.
The correct fix lives inconfig/yahboom_bridge.yaml: invert_vx: true, invert_vy: true. The URDF stays in body convention (D435 = +X). The bridge node sees /cmd_vel in body frame and writes the inverted command to the firmware. Result: Nav2's world view is consistent, the rover obeys correctly.
Why rotating base_link breaks thingsNav2's costmap origin tracks base_link. Rotating base_link 180° rotates the costmap 180°. Then "in front of the rover" in Nav2's frame is behind the rover physically. The local planner picks paths going backward to reach forward goals. Inflation zone appears mirrored. Everything downstream is wrong by π.
camera_name prefix (D435)launch/d435.launch.py
runtime intermediate camera_name: d435_front | d435_rear
realsense2_camera publishes a sub-tree of optical frames under whatever you set camera_name to. To make d435_front_color_optical_frame attach to d435_front_link (defined in URDF), set camera_name:=d435_front when launching. Mismatch the prefix and the frames float free — RViz shows two disconnected trees.
# launch/d435.launch.py realsense_node = Node( package='realsense2_camera', parameters=[{ 'camera_name': 'd435_front', # MUST match the URDF link prefix ... }] )
Test planros2 run tf2_tools view_frames.py after launch. Verify d435_front_link → d435_front_color_optical_frame exists. If camera_color_optical_frame appears instead (no prefix), the camera_name parameter didn't propagate — check the launch include.

8 · Adding a New Platform

Porting the rig to a Go2, R2D2, Roboscout, or any new chassis follows the same recipe: write a platform xacro that defines base_link + chassis visual + the joint to sensor_plate, include sensors_common.urdf.xacro, measure the plate offset, add a PLATFORM_BRIDGES entry. Six concrete steps below.

new-platform checklisturdf/<platform>.urdf.xacro + launch/slam.launch.py
platform advanced 6 files to touch · 1 measurement to make
Concrete recipe for adding (say) Go2 support. Each step is a single small action; the order matters because later steps test earlier ones.
# 1. Copy mecanum.urdf.xacro to urdf/go2.urdf.xacro cp urdf/mecanum.urdf.xacro urdf/go2.urdf.xacro # 2. In go2.urdf.xacro: # - change <robot name="mecanum_x3"> → <robot name="go2"> # - measure base_link → sensor_plate top with rover standing # - update xacro:property base_link_to_plate_top # - replace chassis box dims with Go2 dimensions (or load a mesh) # 3. Add PLATFORM_BRIDGES entry in launch/slam.launch.py: 'go2': (0.0, 0.0, -(BASE_TO_PLATE + 0.036610), 0.0, 0.0, 0.0), # 4. Sanity check: xacro urdf/go2.urdf.xacro > /tmp/go2.urdf # (should emit clean XML, no errors) # 5. Launch: ros2 launch slam_bringup perception.launch.py platform:=go2 # Open RViz, add RobotModel + TF. Confirm all frames present. # 6. Launch SLAM: ros2 launch slam_bringup slam.launch.py platform:=go2 # Map a 5 m hallway. Click floor in RViz — Z must be 0 m ± 5 mm.
If step 5 shows missing framesros2 run tf2_tools view_frames.py → open the PDF → check for orphans. Most common: forgot xacro:include sensors_common.urdf.xacro, or the platform xacro's joint base_link_to_sensor_plate has wrong parent/child names.
If step 6 shows wrong ZBridge Z and base_link_to_plate_top disagree (the two-place update rule). Or you measured the plate offset with the rover crouched but launched while standing (Go2 height varies).
Test planPark rover next to known wall, mark a tape arrow on the floor pointing the rover's "forward" direction. Drive 1 m forward via joystick. In RViz the rover icon should also move +1 m in +X. If it moves in +Y or -X, your URDF rpy is wrong (most likely yaw); fix in URDF, not in the bridge unless the cmd_vel sign is the actual issue (see the mecanum card).

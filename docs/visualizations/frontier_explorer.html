Frontier Explorer

Autonomous frontier-based exploration for the SLAM stack. The robot identifies unexplored boundaries on the occupancy grid, drives toward them via Nav2, and repeats until a time limit expires or no frontiers remain. Then it saves the map and returns home.

How Frontier Exploration Works
Unknown
Free
Occupied / Wall
Frontier
Robot
Path
1
Scan the occupancy grid for frontier cells — free cells adjacent to at least one unknown cell.
2
Cluster adjacent frontier cells into groups and discard clusters smaller than min_frontier_size.
3
Rank clusters by distance cost × potential_scale versus information gain × gain_scale.
4
Send a NavigateToPose goal to the best-ranked frontier via Nav2.
5
Re-evaluate every 3 seconds — if a better frontier appears or the current one is reached, the loop restarts from step 1.
Exploration Parameters
min_frontier_size
config/explore_params.yaml → explore_lite
Minimum frontier cluster size in meters. Clusters smaller than this are ignored. Default: 0.75 m
Explore
Observation: Start at 0.75. If the robot wastes time on tiny gaps between furniture, raise to 1.0–1.5. If it misses doorways, lower to 0.5.
potential_scale
config/explore_params.yaml → explore_lite
Weight for distance cost — higher values make the robot prefer closer frontiers. Default: 3.0
Explore
Observation: At 3.0, the robot strongly favors nearby frontiers (room-by-room clearance). Lower to 1.0 for more adventurous jumps across the map.
gain_scale
config/explore_params.yaml → explore_lite
Weight for information gain — higher values make the robot prefer larger frontiers. Default: 1.0
Explore
Observation: Balance with potential_scale. If the robot ignores large unexplored areas in favor of tiny nearby ones, raise gain_scale or lower potential_scale.
planner_frequency
config/explore_params.yaml → explore_lite
How often frontiers are re-evaluated. Default: 0.33 Hz (~3 s)
Explore
Observation: 3 s is a good default. On slower hardware (Jetson Orin Nano), raising to 5 s (0.2 Hz) reduces CPU load with minimal impact on exploration speed.
progress_timeout
config/explore_params.yaml → explore_lite
Seconds without navigation progress before the current goal is abandoned and a new frontier is selected. Default: 30.0 s
Explore
Observation: If the robot gets stuck spinning at a goal, lower to 15–20. If it gives up too quickly on valid but slow paths, raise to 45.
time_limit_minutes
config/explore_params.yaml → explore_manager
Exploration session duration in minutes. Set to 0 for no time limit (coverage-only). Default: 15.0
Explore
Observation: 15 min covers a typical 2-bedroom apartment. Large houses may need 30–45 min. Use time_limit:=0 for thorough coverage of unknown spaces.
frontier_done_patience
config/explore_params.yaml → explore_manager
Seconds of zero frontiers before declaring exploration complete. Guards against false stops during SLAM loop closures that temporarily clear all frontiers. Default: 30.0 s
Explore
Observation: 30 s handles most RTABMap loop closure transients. If exploration stops prematurely in large spaces, raise to 45–60.
status_publish_rate
config/explore_params.yaml → explore_manager
Rate at which /explore/status JSON is published. Default: 0.2 Hz (every 5 s)
Explore
Observation: 5 s is fine for Foxglove monitoring. If you need real-time frontier count in a custom UI, raise to 1.0 Hz.

Resume Mode

3D Resume (RTABMap)

  • ./start_explore.sh resume:=true
  • Sets delete_db_on_start:=false
  • RTABMap loads existing ~/.ros/rtabmap.db
  • Continues mapping new areas (localization:=false)
  • explore_lite finds remaining frontiers

2D Resume (slam_toolbox)

  • ./start_explore_2d.sh resume:=true map_file:=~/maps/explore_latest
  • Deserializes saved .data + .posegraph
  • Continues in async mapping mode
  • New areas stitched onto existing graph
Constraint: Robot must start in or near already-mapped territory. SLAM needs visual / scan overlap with the prior map to re-localize before new frontiers can be planned.

Where the memory actually goes

The occupancy grid is just 1 byte per cell — even a 200 m map is only 16 MB. The real memory consumers are the SLAM backends and Nav2 costmap layers.

ROS2 + Nav2 base
~500 MB fixed overhead
Nav2 global costmap
3× grid size (static + obstacle + inflation)
RTABMap (3D path)
1–4 GB — 3D point clouds, visual features, loop closure. The bottleneck. ~2–3 MB per keyframe.
slam_toolbox (2D path)
200–800 MB — scan-matching graph only. Much lighter than RTABMap.
FAST-LIO2
~300 MB — IMU + point cloud state estimation
Nav2 local costmap
~2 MB fixed (rolling 6 × 6 m window)
Practical guidance

How far you can push each SLAM backend on the Jetson Orin Nano Super (8 GB).

3D path (FAST-LIO2 + RTABMap): RTABMap stores ~2–3 MB per keyframe (3D clouds + visual features). After ~2000–3000 keyframes, it starts moving old nodes to Long-Term Memory (disk). This corresponds to a ~200–300 m travel path. Map area: ~50 × 50 m (164 × 164 ft / ~27,000 sq ft) comfortably, up to ~100 × 100 m (328 × 328 ft / ~108,000 sq ft) with LTM active.
2D path (slam_toolbox): Only stores 2D laser scans — dramatically lighter. ~100 × 100 m (328 × 328 ft / ~108,000 sq ft) comfortably, up to ~200 × 200 m (656 × 656 ft / ~431,000 sq ft) before the scan graph gets sluggish.
For a typical house: A large single-family home is roughly 15 × 20 m (50 × 65 ft / ~3,200 sq ft). A multi-story house mapped floor-by-floor is ~30 × 30 m (100 × 100 ft / ~9,700 sq ft) per floor. Both are well within the comfortable zone on either path.
FAQ
Does starting position matter?
No. In mapping mode, the power-on position becomes (0, 0, 0) in the map frame. SLAM initializes from wherever the robot is.
What if it gets stuck?
Nav2 recovery chain: spin → backup → wait → clear costmap. If all fail, explore_lite picks the next frontier.
Can it explore multiple rooms?
Yes, automatically. Frontiers are global across the map. When one room is done, the next frontier is in another room.
How do I know when it's done?
Monitor /explore/status topic in Foxglove. Frontier count trending to zero means almost done.
Can I take over with RC mid-explore?
Yes. explore_lite pauses when it can't send goals (Nav2 is busy with your RC commands). When you release, it resumes.
What maps are saved?
3D: ~/maps/explore_YYYYMMDD_HHMMSS.db. 2D: ~/maps/explore_YYYYMMDD_HHMMSS.data + .posegraph.

ROS Topic Guide

Every piece of data flowing through your SLAM stack is a ROS topic — a named stream of messages that one node publishes and others subscribe to. This page catalogues every topic in the project, explains what hardware or algorithm produces it, who consumes it, and exactly how to view it in RViz2 and Foxglove.

The vertical flow diagram below shows how data moves from physical sensors at the top through processing, SLAM, and navigation down to motor commands at the bottom. Click any topic node for details, or scroll to the full reference cards.

Topic Reference Cards
Show:
Raw Sensor Topics
/livox/lidar raw
sensor_msgs/PointCloud2  (or livox_ros_driver2/CustomMsg when xfer_format=1)
The primary 3D point cloud from the Livox Mid-360 LiDAR. Each message contains one scan cycle (~100 ms) of 3D range returns in the sensor's body frame. The Mid-360's non-repetitive scan pattern means each frame covers a different slice of its 360°×59° FOV, building up density over ~0.5 s.
Produced by
Livox ROS Driver 2 — reads UDP packets from the Mid-360 (IP 192.168.1.202). Config: config/mid360.json.
Consumed by
FAST-LIO2 (3D SLAM) — tightly coupled with IMU for odometry
pointcloud_to_laserscan (2D path) — flattens to /scan
RViz / Foxglove — "LiDAR (live)" display
View in RViz2
Add PointCloud2 display → Topic: /livox/lidar. Set Reliability to Best Effort. Style: Points, Size: 1px, Color: FlatColor (grey) or AxisColor for Z-height. Off by default in all configs — enable manually to inspect raw returns.
View in Foxglove
Add a 3D Panel → enable /livox/lidar in the topic sidebar. Set point size to 1-2. Works with the PointCloud2 format only (xfer_format=0); CustomMsg requires conversion.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
Mid-360 UDP → /livox/lidar → FAST-LIO2 → /cloud_registered, /Odometry
Mid-360 UDP → /livox/lidar → pointcloud_to_laserscan → /scan
/livox/imu raw
sensor_msgs/Imu
Raw IMU data from the Mid-360's built-in accelerometer + gyroscope. The linear acceleration values are in g units (not m/s²), which violates REP-145. This topic is never consumed directly by SLAM — it passes through the unit conversion node first.
Produced by
Livox ROS Driver 2 — same driver as the point cloud. IMU runs at ~200 Hz.
Consumed by
imu_units_g_to_ms2 node — converts g→m/s² and republishes as /livox/imu_ms2.
View in RViz2
RViz has no built-in IMU display. Use ros2 topic echo /livox/imu in a terminal, or install the rviz_imu_plugin package for an orientation overlay.
View in Foxglove
Add a Plot Panel → subscribe to /livox/imu.linear_acceleration.x (and y, z). Shows the raw accel waveforms. Also available in a Raw Messages Panel for full struct inspection.
QoS
Reliable · Volatile · Keep Last · Depth 10
Data Chain
Mid-360 IMU → /livox/imu (g) → imu_units_node → /livox/imu_ms2 (m/s²) → FAST-LIO2
/d435_front/camera/color/image_raw raw
sensor_msgs/Image  +  .../color/camera_info (sensor_msgs/CameraInfo)
The RGB color image from the front-facing RealSense D435. Published as a pair: the image itself and a camera_info message with intrinsic calibration (focal length, distortion). RTABMap uses this for visual loop closure — recognising previously-seen places by matching image features.
Produced by
realsense2_camera_node — Intel RealSense ROS wrapper. Runs at 30 fps, 640×480 by default.
Consumed by
RTABMap — remapped as rgb/image and rgb/camera_info for visual feature extraction and loop closure detection.
View in RViz2
Add an Image display → Topic: /d435_front/camera/color/image_raw. Opens a 2D image window inside RViz. Alternatively add a Camera display to see the image overlaid on the 3D view.
View in Foxglove
Add an Image Panel → select the color topic. Foxglove renders the stream at full frame rate. Use side-by-side layout with a 3D panel for context.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
D435 sensor → /d435_front/.../image_raw → RTABMap (feature extraction + loop closure) → map correction
/d435_front/camera/aligned_depth_to_color/image_raw raw
sensor_msgs/Image (16UC1 — depth in mm)
The depth image aligned to the color camera frame. Each pixel is a 16-bit unsigned int representing distance in millimeters. "Aligned" means every depth pixel corresponds 1:1 to the color image — so RTABMap can read color and depth from the same pixel coordinate. Only published when align_depth:=true (SLAM mode).
Produced by
realsense2_camera_node with align_depth:=true. The on-chip alignment costs ~5% CPU on Jetson. Also publishes .../camera_info.
Consumed by
RTABMap — remapped as depth/image. Used to build the 3D structure behind each visual feature, enabling depth-aware ICP refinement.
View in RViz2
Add an Image display → Topic: the aligned depth topic. RViz auto-normalizes 16UC1 into a greyscale image (near = bright, far = dark). Useful for checking depth fill rate and holes.
View in Foxglove
Add an Image Panel → select the depth topic. Foxglove shows depth as a colorized heatmap by default. Switch to "mono" for raw greyscale.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
D435 IR stereo → on-chip depth → alignment → aligned_depth_to_color/image_raw → RTABMap (3D reconstruction)
/imu/data raw
sensor_msgs/Imu
IMU data from the WitMotion WT901C — a standalone 9-axis IMU mounted on the sensor rig. Provides acceleration, angular velocity, and fused orientation (roll/pitch/yaw). Currently used as a reference/diagnostic source; the primary SLAM IMU is the Mid-360's built-in unit. Available for future EKF fusion if needed.
Produced by
wt901c_imu_node — custom parser for WitMotion 0x61 protocol over USB-serial (/dev/ttyUSB0, 115200 baud). Configurable 10-200 Hz.
Consumed by
Currently no active consumer in the main stack. Available for EKF sensor fusion or attitude reference.
View in RViz2
Install rviz_imu_plugin → add an Imu display → Topic: /imu/data. Shows an orientation box on the robot. Or use ros2 topic echo for raw values.
View in Foxglove
Plot Panel to graph accel/gyro axes over time. 3D Panel can show the orientation quaternion as a pose if you also publish a PoseStamped.
QoS
Default (Reliable · Keep Last · Depth 50)
Data Chain
WT901C serial → wt901c_imu_node → /imu/data → (available for EKF / diagnostics)
Processed & Converted Topics
/livox/imu_ms2 processed
sensor_msgs/Imu
The Mid-360 IMU data after unit conversion — linear acceleration converted from g to m/s² (×9.80665) per REP-145. Angular velocity and orientation are passed through unchanged. The node auto-detects on the first message whether conversion is needed (some firmware versions already output m/s²).
Produced by
imu_units_g_to_ms2 node. Subscribes to /livox/imu (Best Effort), publishes as Reliable for downstream consumers.
Consumed by
FAST-LIO2 — used for IMU pre-integration and tightly-coupled lidar-inertial odometry. This is the critical IMU input for 3D SLAM.
View in RViz2
Same as /livox/imu — use rviz_imu_plugin or ros2 topic echo. The values should now read ~9.8 m/s² on the gravity axis when stationary.
View in Foxglove
Plot Panel — plot .linear_acceleration.z and verify it reads ~9.81 at rest. Good sanity check that the conversion is working.
QoS
Reliable · Volatile · Keep Last · Depth 10
Data Chain
/livox/imu (g) → imu_units_node (×9.80665) → /livox/imu_ms2 (m/s²) → FAST-LIO2
/scan processed
sensor_msgs/LaserScan
A 2D horizontal ring of range measurements — the 3D point cloud collapsed into a single plane. Produced by pointcloud_to_laserscan, which takes all 3D points within a Z band (scan_z_min to scan_z_max) and projects them into a 360° range array. The input source differs by mode: /livox/lidar in the 2D path, /cloud_registered_body in the 3D nav path.
Produced by
pointcloud_to_laserscan node.
2D SLAM: input = /livox/lidar
3D Nav: input = /cloud_registered_body
Consumed by
rf2o_laser_odometry — laser-only odom (2D path)
slam_toolbox — 2D grid mapping
Nav2 costmap — obstacle layer (both paths)
View in RViz2
Add a LaserScan display → Topic: /scan. Set Reliability to Best Effort. Style: Points, Size 3-4px, FlatColor cyan. Shows a ring of dots around the robot — the wall contour at the scan height.
View in Foxglove
3D Panel renders the LaserScan as a ring of points. You can also add a Plot Panel and graph /scan.ranges to see range-vs-angle for debugging specific dead sectors.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
/livox/lidar or /cloud_registered_body → pointcloud_to_laserscan (Z-band filter + projection) → /scan → rf2o / slam_toolbox / Nav2 costmap
/d435_front/camera/depth/color/points processed
sensor_msgs/PointCloud2 (with RGB fields)
An organized RGB-D point cloud computed from the D435's depth + color images. Each point carries XYZ position and RGB color. Covers the camera's ~87°×58° FOV. This is the RealSense driver's on-board depth-to-pointcloud conversion. Disabled in SLAM mode (pointcloud:=false) because RTABMap builds its own cloud internally — enable it in raw/perception mode for direct visualization.
Produced by
realsense2_camera_node with pointcloud:=true. CPU-intensive (~15% on Jetson); only enable when needed.
Consumed by
RViz / Foxglove only — pure visualization. RTABMap reads the depth image directly, not this point cloud.
View in RViz2
Add a PointCloud2 display → Topic: the depth/color/points topic. Set Color Transformer to RGB8 to see the real colors. Size: 2px. Best Effort reliability.
View in Foxglove
3D Panel → enable the topic. Foxglove renders the RGB colors automatically. Great for side-by-side comparison with the lidar cloud to check extrinsic alignment.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
D435 IR stereo → depth image + color image → RealSense driver (depth→cloud) → .../depth/color/points → RViz / Foxglove
SLAM & Odometry Outputs
/Odometry slam
nav_msgs/Odometry
IMU-fused lidar-inertial odometry from FAST-LIO2. This is the robot's estimated 6-DOF pose (position + orientation) and velocity in the camera_init frame. Updated at ~10 Hz. FAST-LIO2 tightly couples the Mid-360's point cloud with the corrected IMU to produce drift-resistant odometry that serves as the backbone of the 3D SLAM pipeline.
Produced by
FAST-LIO2 (fastlio_mapping node). Inputs: /livox/lidar + /livox/imu_ms2. Also publishes TF: camera_init → body.
Consumed by
RTABMap — remapped as odom; used as the odometry prior for graph optimization
Nav2 controller_server — robot velocity feedback
RViz — Odometry display (red arrows)
View in RViz2
Add an Odometry display → Topic: /Odometry. Set Reliability to Best Effort. Shape: Arrow, Color: red, Keep: 100. Draws a trail of arrows showing the robot's path.
View in Foxglove
3D Panel renders the odometry as arrow markers. Add a Plot Panel to graph /Odometry.twist.twist.linear.x for velocity over time — great for spotting jerks or drift.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
/livox/lidar + /livox/imu_ms2 → FAST-LIO2 (iterated EKF) → /Odometry → RTABMap / Nav2 / RViz
/odom slam
nav_msgs/Odometry
Laser-only 2D odometry from rf2o_laser_odometry. Estimates the robot's 2D pose (x, y, yaw) by scan-matching consecutive /scan frames. Used exclusively in the 2D SLAM path as the odometry source for slam_toolbox. Lighter weight than FAST-LIO2 but 2D-only and more drift-prone without IMU fusion.
Produced by
rf2o_laser_odometry node. Input: /scan. Also publishes TF: odom → base_link (when publish_tf:=true).
Consumed by
slam_toolbox — odometry input for pose-graph SLAM
Nav2 — via TF tree (odom → base_link)
RViz — Odometry display (green arrows)
View in RViz2
Add an Odometry display → Topic: /odom. Set Reliability to Reliable (rf2o publishes Reliable, unlike FAST-LIO). Shape: Arrow, Color: green, Keep: 100.
View in Foxglove
Same approach as /Odometry — 3D Panel for the arrow trail, Plot Panel for velocity graphs.
QoS
Reliable · Volatile · Keep Last · Depth 100
Data Chain
/livox/lidar → pointcloud_to_laserscan → /scan → rf2o (scan matching) → /odom → slam_toolbox / Nav2
/cloud_registered slam
sensor_msgs/PointCloud2
The accumulated world-frame point cloud from FAST-LIO2 — every scan stitched together in the camera_init (world) frame. This grows continuously as the robot drives, forming a dense 3D model of the environment. Fed into the viz_clip node for ceiling-removal before display. Also the basis for RTABMap's 3D map overlay.
Produced by
FAST-LIO2 — outputs the current map after each lidar-inertial update cycle.
Consumed by
viz_clip_node → clips Z range → publishes as /cloud_viz_clipped. Also viewable raw in RViz.
View in RViz2
Add a PointCloud2 display → Topic: /cloud_registered. Best Effort. AxisColor on Z with rainbow. Usually you view /cloud_viz_clipped instead (ceiling removed), but this shows the full unfiltered cloud.
View in Foxglove
3D Panel → subscribe to the topic. Color by position.z. Large clouds may need point decimation in Foxglove settings for smooth rendering.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
FAST-LIO2 → /cloud_registered (world frame) → viz_clip_node → /cloud_viz_clipped
/cloud_registered_body slam
sensor_msgs/PointCloud2
The current scan registered in the body frame — same data as /cloud_registered but expressed relative to the robot, not the world. This is what RTABMap uses for ICP (Iterative Closest Point) refinement, and what pointcloud_to_laserscan uses in the 3D nav path to produce /scan.
Produced by
FAST-LIO2 — body-frame output of the current scan, post-registration.
Consumed by
RTABMap — remapped as scan_cloud for ICP loop closure
pointcloud_to_laserscan (3D nav path) — flattens to /scan
View in RViz2
Add a PointCloud2 display → Topic: /cloud_registered_body. Best Effort. This cloud moves with the robot — useful for checking what the current frame looks like before accumulation.
View in Foxglove
3D Panel — the cloud appears attached to the robot since it's in body frame.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
FAST-LIO2 → /cloud_registered_body → RTABMap (ICP) + pointcloud_to_laserscan → /scan
/cloud_viz_clipped slam
sensor_msgs/PointCloud2
The accumulated world cloud with ceiling and floor points removed by Z-axis clipping. Parameters z_min (default -3.0 m) and z_max (default 3.0 m) in the camera_init frame control the band. Visualization-only — does not affect SLAM or navigation. Makes the 3D view much cleaner by removing ceiling geometry that would otherwise obscure the map from an orbit camera.
Produced by
viz_z_clip node. Input: /cloud_registered. Parameters are runtime-tunable via ros2 param set.
Consumed by
RViz / Foxglove only — the "Cloud (world)" display in all 3D configs.
View in RViz2
This is the default Cloud (world) display in slam.rviz and nav.rviz. PointCloud2, AxisColor on Z, rainbow, 2px points. The main visual during 3D SLAM — watch the map grow.
View in Foxglove
3D Panel → /cloud_viz_clipped. Color by position.z. Same experience as RViz but works over Foxglove Bridge from a remote workstation.
QoS
Best Effort · Volatile · Keep Last · Depth 5
Data Chain
/cloud_registered → viz_z_clip (strip z < -3 and z > 3) → /cloud_viz_clipped → RViz / Foxglove
/map slam
nav_msgs/OccupancyGrid
The 2D occupancy grid — the floorplan that SLAM builds. Each cell is a probability 0-100 (free → occupied) or -1 (unknown). Black = wall, white = free, grey = unexplored. Produced by RTABMap (3D path) or slam_toolbox (2D path). This is the persistent spatial knowledge Nav2 uses for global planning. Uses Transient Local durability so late-joining subscribers (like RViz started after SLAM) receive the last published map immediately.
Produced by
RTABMap (3D) or slam_toolbox (2D). Updated when the robot explores new areas or loop closures correct the graph.
Consumed by
Nav2 global_costmap — static layer reads this as the base obstacle data
Nav2 planner_server — plans paths through free cells
RViz / Foxglove — the Map display
View in RViz2
Add a Map display → Topic: /map. Set Reliability to Reliable, Durability to Transient Local. Color Scheme: map. Alpha: 0.7, Draw Behind: true. This is the background floorplan everything else renders on top of.
View in Foxglove
Use a dedicated Map Panel for a top-down 2D view (great for seeing the full floorplan). Also renders in the 3D Panel as a floor texture. Foxglove handles Transient Local QoS automatically.
QoS
Reliable · Transient Local · Keep Last · Depth 1
Data Chain
3D: /livox/lidar + /livox/imu_ms2 → FAST-LIO2 → /cloud_registered_body + /Odometry + D435 images → RTABMap → /map
2D: /livox/lidar → pointcloud_to_laserscan → /scan → slam_toolbox → /map
Navigation Topics (Nav2)
/global_costmap/costmap nav
nav_msgs/OccupancyGrid
The inflated global obstacle map — Nav2 takes the raw /map and expands every occupied cell by the robot's inscribed radius plus an inflation zone. The result is a cost gradient: lethal near walls, decreasing cost with distance, zero in open space. The global planner uses this to find the cheapest (safest) route from A to B. Covers the entire known environment.
Produced by
Nav2 global_costmap node. Reads: /map (static layer) + /scan (obstacle layer). Robot radius: 0.4 m.
Consumed by
Nav2 planner_server (NavFn/Smac) — plans the global path through inflated-free space. RViz / Foxglove for visualization.
View in RViz2
Add a Map display → Topic: /global_costmap/costmap. Reliable + Transient Local. Color Scheme: costmap (rainbow inflation gradient). Alpha: 0.5. Overlays the inflation zones on top of the base map.
View in Foxglove
3D Panel → enable the costmap topic. Use "costmap" color map. Toggle on/off to compare with the base map underneath.
QoS
Reliable · Transient Local · Keep Last · Depth 1
Data Chain
/map (static layer) + /scan (obstacle layer) → Nav2 global_costmap (inflation) → /global_costmap/costmap → planner_server → /plan
/local_costmap/costmap nav
nav_msgs/OccupancyGrid
A rolling-window costmap centered on the robot — typically 3-5 m in each direction. Updates rapidly from live /scan data to detect new obstacles the static map doesn't know about (people, doors, furniture that moved). The local controller reads this for real-time collision avoidance. Unlike the global costmap, this one doesn't depend on the map being correct — it trusts the live sensor.
Produced by
Nav2 local_costmap node. Reads: /scan (obstacle layer). Publishes at ~5 Hz, rolling-window geometry.
Consumed by
Nav2 controller_server (DWB/MPPI) — scores candidate trajectories against this costmap for reactive obstacle avoidance.
View in RViz2
Add a Map display → Topic: /local_costmap/costmap. Same settings as global costmap. You'll see a small colored square that moves with the robot — obstacles within it light up in real time.
View in Foxglove
3D Panel — renders identically to the global costmap but as a smaller rolling patch. Useful to toggle independently from the global costmap to reduce visual clutter.
QoS
Reliable · Transient Local · Keep Last · Depth 1
Data Chain
/scan (live sensor) → Nav2 local_costmap (obstacle + inflation layers) → /local_costmap/costmap → controller_server → /cmd_vel
/plan nav
nav_msgs/Path
The global navigation path — a sequence of poses from the robot's current position to the goal. Computed by the Nav2 global planner (NavFn or Smac) over the global costmap. Drawn as a green line in RViz. Recomputes periodically or when the robot deviates too far. If this path takes a weird detour, the problem is in the global costmap (inflation too aggressive, obstacle where there isn't one).
Produced by
Nav2 planner_server. Triggered by /goal_pose via the behavior tree navigator.
Consumed by
Nav2 controller_server — follows this path with the local controller. RViz / Foxglove — the green path overlay.
View in RViz2
Add a Path display → Topic: /plan. Reliable QoS. Color: green (25, 255, 0), Line Width: 0.05. The classic green planning line.
View in Foxglove
3D Panel renders Path messages as lines automatically. Color customizable in panel settings.
QoS
Reliable · Volatile · Keep Last · Depth 5
Data Chain
/goal_pose → bt_navigator → planner_server + /global_costmap/costmap → /plan → controller_server
/local_plan nav
nav_msgs/Path
The short-horizon trajectory the local controller is executing right now — typically 1-3 seconds of lookahead. The controller (DWB or MPPI) samples hundreds of candidate trajectories, scores each against the local costmap and the global plan, and picks the best. This blue line shows you what the robot is about to do. If the robot oscillates, this line will flicker between alternatives.
Produced by
Nav2 controller_server (DWB or MPPI plugin). Updated at the controller frequency (~20 Hz).
Consumed by
RViz / Foxglove only — pure visualization. The controller internally converts this to /cmd_vel.
View in RViz2
Add a Path display → Topic: /local_plan. Reliable QoS. Color: blue (0, 0, 255), Line Width: 0.04. A short blue curve that updates in real time.
View in Foxglove
3D Panel → enable /local_plan. Blue line. Compare against /plan (green) to see how closely the local controller is tracking the global path.
QoS
Reliable · Volatile · Keep Last · Depth 5
Data Chain
/plan + /local_costmap/costmap + /Odometry → controller_server (trajectory scoring) → /local_plan (viz) + /cmd_vel (action)
Control & Interaction Topics
/cmd_vel control
geometry_msgs/Twist
The velocity command sent to the chassis — linear.x (forward/back), linear.y (strafe, mecanum only), angular.z (rotation). This is the final output of the entire perception-planning-control pipeline. The yahboom_bridge_node reads it and sends serial motor commands to the YB-ERF01 STM32. Safety clamped to max_vx=0.5 m/s, max_vy=0.3 m/s, max_wz=1.0 rad/s. A 0.5 s watchdog stops the motors if no new message arrives.
Produced by
Nav2 controller_server (autonomous), teleop_twist_keyboard (manual), or any other Twist publisher.
Consumed by
yahboom_bridge_node — converts Twist to Rosmaster_Lib serial protocol → STM32 → motor drivers → wheels. Note: SBUS RC silently overrides this — power off the Taranis when using /cmd_vel.
View in RViz2
No built-in display. Use ros2 topic echo /cmd_vel in a terminal. Or install rviz_2d_overlay_plugins for a velocity gauge HUD overlay.
View in Foxglove
Plot Panel — graph /cmd_vel.linear.x, .linear.y, .angular.z over time. Shows the commanded velocity profile. Great for tuning controller gains.
QoS
Reliable · Volatile · Keep Last · Depth 10
Data Chain
Nav2 controller_server → /cmd_vel → yahboom_bridge_node (serial) → STM32 → motor drivers → wheels
/goal_pose control
geometry_msgs/PoseStamped
The navigation goal — where you want the robot to go. Published when you click the "2D Goal Pose" tool in RViz and drag an arrow on the map. The arrow direction sets the desired final orientation. Nav2's behavior tree navigator receives this and kicks off the full plan-follow-recover cycle.
Produced by
RViz "2D Goal Pose" tool (click + drag on the map). Also publishable via CLI: ros2 topic pub /goal_pose geometry_msgs/PoseStamped ...
Consumed by
Nav2 bt_navigator — triggers the navigation behavior tree which calls the planner, then the controller, with recovery behaviors on failure.
View in RViz2
This is an input tool, not a display — you create it by selecting the "2D Goal Pose" button in the toolbar, then clicking on the map. The goal appears as a green arrow.
View in Foxglove
Foxglove's 3D Panel supports "Publish" mode — click to publish a PoseStamped on a configured topic. Set the publish topic to /goal_pose in the panel settings.
QoS
Reliable · Volatile · Keep Last · Depth 1
Data Chain
Human click → RViz/Foxglove → /goal_pose → bt_navigator → planner_server → /plan → controller_server → /cmd_vel
/initialpose control
geometry_msgs/PoseWithCovarianceStamped
A manual pose hint — tells RTABMap (or AMCL) "the robot is approximately here." Published when you click the "2D Pose Estimate" tool in RViz and drag on the map. Use this when the robot is lost (localization jumped or the map doesn't match). RTABMap re-seeds its localization search from this pose.
Produced by
RViz "2D Pose Estimate" tool — click + drag to set the robot's approximate position and heading.
Consumed by
RTABMap (3D localization mode) or slam_toolbox (2D localization mode) — re-initializes the pose estimate for relocalization.
View in RViz2
Input tool — click "2D Pose Estimate" in the toolbar, then click + drag on the map. The arrow represents position + heading.
View in Foxglove
Same publish mechanism as /goal_pose — configure a 3D Panel publish topic to /initialpose. Note: Foxglove publishes PoseStamped, not PoseWithCovarianceStamped — some localization nodes may need adaptation.
QoS
Reliable · Volatile · Keep Last · Depth 1
Data Chain
Human click → RViz/Foxglove → /initialpose → RTABMap / slam_toolbox (relocalize) → corrected TF + /map
System & Transform Topics
/tf system
tf2_msgs/TFMessage
The dynamic transform tree — constantly-updating coordinate frame relationships. Multiple nodes publish here simultaneously: FAST-LIO2 publishes camera_init → body, RTABMap publishes map → camera_init, rf2o publishes odom → base_link, and the yahboom_bridge publishes body → base_link. Every other display depends on TF to know where data belongs in space.
Produced by
Every node that knows where something is — FAST-LIO2, RTABMap, rf2o, robot_state_publisher, static_transform_publisher, Nav2 (camera_init→odom alias).
Consumed by
Everything. Every RViz display, Nav2, RTABMap, pointcloud_to_laserscan — all use TF to transform data between frames.
View in RViz2
Add a TF display. Shows colored axes at every frame. Show Names: true to label them. If any display shows "Transform [x] does not exist," the problem is a missing TF publisher.
View in Foxglove
3D Panel shows transform frames in the sidebar under "Transforms." Toggle individual frames on/off. Foxglove also has a Topic Graph panel that visualizes the TF tree structure.

Best Effort · Volatile · Keep Last · Depth 100
Frame Chain (3D path)

/tf_static system
tf2_msgs/TFMessage
Static (non-moving) transforms — published once with Transient Local durability so late-joining subscribers get them immediately. These are the fixed geometric relationships: sensor mount positions on the chassis, camera extrinsics, and URDF joint positions for non-moving joints. Published by robot_state_publisher from the URDF.
Produced by
robot_state_publisher (from URDF), static_transform_publisher launch nodes (explicit extrinsics).
Consumed by
Everything — same as /tf. Static frames are cached permanently by the TF listener.
View in RViz2
Appears automatically in the TF display alongside dynamic frames. Static frames have a "static" tag in the frame details.
View in Foxglove
Same as /tf — appears in the 3D Panel sidebar under Transforms. Foxglove handles the Transient Local durability automatically.
QoS
Reliable · Transient Local · Keep Last · Depth 1
Source
URDF/xacro → robot_state_publisher → /tf_static → all nodes (cached permanently)
/robot_description system
std_msgs/String (URDF XML)
The complete URDF XML describing the robot's geometry — every link, joint, visual mesh, collision shape, and sensor mount. Published once by robot_state_publisher as a latched parameter/topic. RViz reads this to render the RobotModel display. This is the single source of truth for the robot's physical structure.
Produced by
robot_state_publisher — processes the xacro file (e.g. urdf/go2.urdf.xacro or urdf/yahboom.urdf.xacro) at launch, publishes the resulting XML.
Consumed by
robot_state_publisher itself — derives the static TF tree from the URDF joints
RViz RobotModel display — renders the meshes
Nav2 — reads the robot footprint
View in RViz2
Add a RobotModel display → Description Topic: /robot_description. Reliability: Reliable, Durability: Transient Local. Renders the full robot mesh positioned by TF.
View in Foxglove
3D Panel automatically picks up /robot_description and renders the URDF. Toggle in the panel sidebar under "URDF." Also viewable as raw XML in a Raw Messages Panel.
QoS
Reliable · Transient Local · Keep Last · Depth 1
Source
xacro files → xacro processor (launch time) → robot_state_publisher → /robot_description + /tf_static
